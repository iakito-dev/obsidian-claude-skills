# log-from-pr

他人のPRや過去のイシューを観察して学びを資産化するスキル。
**ジュニアエンジニアにとってレビュー指摘は金脈** — シニアの設計原則が抽出できる。
出力ノートの形式は `/log-learning` と同じ。**3年後の自分にも価値がある知識**だけを残す。

## Vault
`~/obsidian-vault/`（ローカル限定）

```
obsidian-vault/
├── 00_Inbox/
├── 10_Issues/               # 作業記録（<ISSUE-ID>.md）
├── 20_Knowledge/            # 学びの蓄積（技術ドメイン別）
│   ├── 01_フロントエンド/
│   ├── 02_バックエンド/
│   ├── 03_データベース/
│   ├── 04_インフラ/
│   ├── 05_開発ツール/
│   ├── 06_AI駆動開発/
│   ├── 07_アーキテクチャ/
│   └── 08_CS基礎・Web開発/
└── 90_Templates/
```

新規ノート作成時は適切なカテゴリフォルダに配置し、`category` フロントマターを記録する。
**配置判断は Vault README の「境界判断のルール」と「早見表」を参照する**。

## 入力
- GitHub PR URL（例: `https://github.com/<org>/<repo>/pull/123`）
- イシュートラッカーの URL（紐付くPRを `gh` 経由で自動取得）
- PR番号のみ（リポジトリは cwd から推測）

## 動作

### 1. PR データの取得
```bash
gh pr view <URL or 番号> --json title,body,files,comments,reviews,author,headRefName,baseRefName
gh pr diff <URL or 番号>
```

- リポジトリ判別: URL > cwd の git remote
- イシュー URL の場合: イシュー詳細を取得 → PR リンクを抽出
- 取得失敗時はユーザーに確認

### 2. イシュー番号の抽出
- ブランチ名 / PRタイトル / PR本文から `[A-Z]+-\d+` を抽出（プロジェクトの ID 形式に応じて調整）
- `10_Issues/<ISSUE-ID>.md` が既存なら追記
- 既存ノートが `source: pr-observation` でなければ「上書きせず別名で作りますか?」と確認
- 新規なら `90_Templates/issue-observation.md` から生成

### 3. 学び候補の抽出

PR本文・diff・レビューコメントから以下を順に抽出:

#### a. 適用されている設計パターン（最重要）
- ファイル構造・命名から読み取れるアーキ判断
- 例: 「page.tsx は async（Server Component）、Dialog 部分は 'use client'」
  → [[Server Component と Client Component の境界]]

#### b. レビュー指摘 → 修正の往復（最重要）
- 各レビューコメントについて以下を整理:
  - 指摘内容（一般化した形で。固有名や個人名は記録しない）
  - 修正前 / 修正後の差分
  - **指摘の根拠となる一般原則は何か?** ← ここを必ず抽出
- 例: 「Selectable<...> 型を Repository 外に漏らさないよう指摘」
  → 根拠: Anti-Corruption Layer / レイヤリング

#### c. PR本文の "なぜこうしたか"
- 「影響範囲調査」「テスト戦略」「設計判断」セクションがあれば読む
- ADR / RFC へのリンクがあれば該当ファイルを Read

### 4. 制約: 提案は最大 5 個まで

1つのPRから10個もノートを作らない。**最も学びになる5個**に絞る。
**「3年後の自分・別プロジェクト・別言語でも使えるか?」** で判断する。

優先順位:
1. レビュー指摘から抽出された一般原則（最優先）
2. PR本文に明記された設計判断
3. 非自明な設計パターン
4. その他

**記録しない**:
- 細かいライブラリの罠・workaround（ググれば出るもの）
- ライブラリ固有の関数仕様（compact() の挙動など）

### 5. ユーザー確認 → ノート作成

1個ずつ「これを記録しますか?」と確認。Yes なら:
- `20_Knowledge/XX_カテゴリ/` に新規 or 既存追記
- 作業記録ノート（`source: pr-observation`）に学びを集約

作業記録ノート側のセクション:
- `## 3行サマリ` — 何のPR / どんな対応か / 自分にとっての意味
- `## 対応内容` — 子イシューやPR単位で実装の要点を箇条書き
- `## 今後への学び` — 設計判断 / ハマりどころ / 設計原則。`[[20_Knowledge/...]]` へリンク

### 6. 既存ノートとの整合

新しい知識ノートを作る前に既存 `20_Knowledge/` を grep:
- 既存ノートがあれば「[[XXX]] に "このプロジェクトでの実例" として追記しますか?」
- 概念は新しいが既存に近いものがあれば「[[YYY]] の関連として追加しますか?」

### 7. 未定義リンクの検出
新規ノート作成後、本文中の技術用語を抽出して `20_Knowledge/` を grep し、
ノートが未作成のものを1個ずつ提示して作成を確認する。
No なら飛ばす。

### 8. 出力
最後に作成・更新したファイルのパスを箇条書きで提示。

## ノート構造（`20_Knowledge/`）

**必須**: `## サマリ` (何/なぜ/使い所) / `## 具体的な解説` / `## 担当プロジェクトでの実例` / `## 参考情報`

**オプション**（書くと価値が出る時だけ）:
- `## 似た概念との違い` — 比較対象がある時、比較表で
- `## 使う場面 / 使わない場面` — 過剰適用を防ぐ判断軸

雛形であり、内容に応じて取捨選択する。frontmatter: `mastery: shallow` で開始。
ゴミナレッジを溜めないため、ググれば出る細かい知識は記録しない。

## 記録しないもの
- イシュー / PR本文の丸ごとコピー（URLリンクのみ）
- 指摘者・PR作者以外の個人名（author は frontmatter のみ）
- DB接続情報・トークン・APIキー
- レビュー指摘の感情的・人格的な側面（あれば無視、技術的内容のみ抽出）
- 細かいライブラリ固有の罠（方針外）
