# log-from-pr

他人のPRや過去のイシューを観察して学びを資産化するスキル。
**PR の差分・ファイル構成・PR 本文を咀嚼**して「何を・なぜ・どうやったか」を自分の言葉でまとめ、
そこから一般化できる設計パターン・原則を `20_Knowledge/` に切り出す。

AI 前提の開発ではレビュー指摘が少ないことが多い。レビュー指摘は **あれば** 拾うが、メインの素材は **diff と PR 本文** にする。

**3年後の自分にも価値がある知識**だけを残す。

## Vault
`~/obsidian-vault/`（ローカル限定）

```
obsidian-vault/
├── 00_Inbox/
├── 10_Issues/               # 作業記録（サイクル別フォルダ）
│   └── cycle-<N>/<ISSUE-ID>.md
├── 20_Knowledge/            # 学びの蓄積（技術ドメイン別）
│   ├── 01_フロントエンド/
│   ├── 02_バックエンド/
│   ├── 03_データベース/
│   ├── 04_インフラ/
│   ├── 05_開発ツール/
│   ├── 06_AI駆動開発/
│   ├── 07_アーキテクチャ/
│   └── 08_CS基礎・Web開発/
└── 90_Templates/
```

新規ノート作成時は適切なカテゴリフォルダに配置し、`category` フロントマターを記録する。
**配置判断は Vault README の「境界判断のルール」と「早見表」を参照する**。

## 入力
- GitHub PR URL（例: `https://github.com/<org>/<repo>/pull/123`）
- イシュートラッカーの URL（紐付くPRを `gh` 経由で自動取得）
- PR番号のみ（リポジトリは cwd から推測）
- 親イシュー URL（複数の子 PR をまとめて咀嚼するパターン）

## 動作

### 1. PR データの取得
```bash
gh pr view <URL or 番号> --json title,body,files,comments,reviews,author,headRefName,baseRefName,mergedAt
gh pr diff <URL or 番号>
```

- リポジトリ判別: URL > cwd の git remote
- イシュー URL の場合: イシュー詳細を取得 → 紐付く PR リンクを抽出
- 親イシューに複数 PR がぶら下がる場合は **全部取得して横断的に咀嚼**
- 取得失敗時はユーザーに確認

### 2. イシュー番号 + サイクルの特定
- ブランチ名 / PRタイトル / PR本文から `[A-Z]+-\d+` を抽出（プロジェクトの ID 形式に応じて調整）
- 作業記録はサイクル別フォルダ `10_Issues/cycle-<N>/` に格納する
- Linear MCP が利用可能なら:
  1. `mcp__linear-server__get_issue` でイシューの `cycleId` を取得
  2. `mcp__linear-server__list_cycles` で該当 `cycleId` の `number` を引く
  3. フォルダ名は `cycle-<number>` (例: `cycle-13`)
- まず `find 10_Issues -name "<ISSUE-ID>.md"` で既存ノートを横断検索（過去サイクルのフォルダに既にあれば移動せず追記）
- 既存ノートが `source: pr-observation` でなければ「上書きせず別名で作りますか?」と確認
- 新規なら `90_Templates/issue-observation.md` から `10_Issues/cycle-<N>/<ISSUE-ID>.md` を生成
- サイクル情報が無い・取得できない場合は `10_Issues/` 直下にフォールバック、またはユーザーに確認

### 3. PR 内容の咀嚼（メイン）

PR 本文・diff・ファイル構成から以下を順に抽出する:

#### a. 何を変更したか（diff / files ベース）
- 変更されたファイル群を眺めて、**作業の輪郭** を掴む
  - 新規追加されたファイル → 何のための新規導入か
  - 既存ファイルの大きな変更 → 何を入れ替えたか
  - 設定ファイル (`*.yml` / `*.tf` / `package.json` 等) の変更 → どの基盤が動いたか
- 子 PR が複数ある場合はそれぞれの **担当範囲** を明確にする（例: 「PR A は terraform 側、PR B はアプリ側」）

#### b. なぜそうしたか（PR 本文ベース）
- 「背景 / Background」「設計判断 / Decisions」「採用構成」セクションを読む
- **チケット文面と実装が乖離している点** があれば必ず拾う（コスト・運用負荷・スコープ調整など、後から見て価値のある判断）
- 影響範囲調査セクションがあれば、どう調査したかを学ぶ
- ADR / RFC へのリンクがあれば該当ファイルを Read

#### c. 適用されている設計パターン
- ファイル構造・命名・モジュール境界から読み取れるアーキ判断
- 例: 「page.tsx は async（Server Component）、Dialog 部分は 'use client'」
  → [[Server Component と Client Component の境界]]
- 例: 「既存モジュールに `additional_*` 系変数で非破壊拡張」
  → [[Infrastructure as Code とモジュール化]]

#### d. ハマりどころ・追加修正 PR
- マージ後の修正 PR / hotfix があれば必ず読む（**本番で初めて気付いた問題**は金脈）
- 「原因 → 修正」をペアで抽出し、一般原則に昇華する

#### e. レビュー指摘（あれば）
- AI 前提だとレビュー指摘は少ないが、**あれば優先度高で拾う**
- 各レビューコメントについて以下を整理:
  - 指摘内容（一般化した形で。固有名や個人名は記録しない）
  - 修正前 / 修正後の差分
  - **指摘の根拠となる一般原則は何か?** ← ここを必ず抽出
- レビューが無くても気にしない。`#d` までで十分な学びが取れることが多い

### 4. 制約: 知識ノートへの切り出しは最大 5 個

1つのPRから10個もノートを作らない。**最も学びになる5個**に絞る。
**「3年後の自分・別プロジェクト・別言語でも使えるか?」** で判断する。

優先順位:
1. ハマりどころ・追加修正 PR から抽出された一般原則
2. PR 本文に明記された設計判断（特にチケットと実装の乖離理由）
3. 非自明な設計パターン
4. レビュー指摘から抽出された原則（あれば）

**記録しない**:
- 細かいライブラリの罠・workaround（ググれば出るもの）
- ライブラリ固有の関数仕様（`compact()` の挙動など）

### 5. ノート作成

作業記録ノート（`10_Issues/<ISSUE-ID>.md`、`source: pr-observation`）のセクション構成は固定:

- `## 3行サマリ` — 何のPR・親イシューか / どんな対応か / 自分にとっての意味（3行程度の自然な日本語）
- `## 対応内容` — 子PR / コミット単位で実装の要点を箇条書き。**チケットと実装の乖離**もここに書く
- `## 今後への学び` — 設計判断 / ハマりどころ / 設計原則。`[[20_Knowledge/...]]` へリンク

知識ノート（`20_Knowledge/XX_カテゴリ/`）は標準テンプレで作る:
- 既存ノートがあれば「[[XXX]] に "このプロジェクトでの実例" として追記しますか?」
- 概念は新しいが既存に近いものがあれば「[[YYY]] の関連として追加しますか?」

### 6. 未定義リンクの検出
新規ノート作成後、本文中の技術用語を抽出して `20_Knowledge/` を grep し、
ノートが未作成のものを1個ずつ提示して作成を確認する。
No なら飛ばす。

### 7. 出力
最後に作成・更新したファイルのパスを箇条書きで提示。

## ノート構造（`20_Knowledge/`）

**必須**: `## サマリ` (何/なぜ/使い所) / `## 具体的な解説` / `## 担当プロジェクトでの実例` / `## 参考情報`

**オプション**（書くと価値が出る時だけ）:
- `## 似た概念との違い` — 比較対象がある時、比較表で
- `## 使う場面 / 使わない場面` — 過剰適用を防ぐ判断軸

雛形であり、内容に応じて取捨選択する。frontmatter: `mastery: shallow` で開始。
ゴミナレッジを溜めないため、ググれば出る細かい知識は記録しない。

## 記録しないもの
- イシュー / PR本文の丸ごとコピー（URLリンクのみ）
- PR作者以外の個人名（author は frontmatter のみ）
- DB接続情報・トークン・APIキー
- レビュー指摘の感情的・人格的な側面（あれば無視、技術的内容のみ抽出）
- 細かいライブラリ固有の罠（方針外）

## トリガー
ユーザーが明示的に呼ぶ前提。Claude が自発的に提案するシーンは無い。
