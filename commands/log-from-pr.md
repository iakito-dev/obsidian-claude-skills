# log-from-pr

他人のPRや過去のイシューを観察して学びを資産化するスキル。
**PR の差分・ファイル構成・PR 本文を咀嚼**して「何を・なぜ・どうやったか」を自分の言葉でまとめ、
そこから一般化できる設計パターン・原則を `20_Knowledge/` に切り出す。

AI 前提の開発ではレビュー指摘が少ないことが多い。レビュー指摘は **あれば** 拾うが、メインの素材は **diff と PR 本文** にする。

**3年後の自分にも価値がある知識**だけを残す。

## Vault
`~/obsidian-vault/`（ローカル限定）

```
obsidian-vault/
├── 00_Inbox/
├── 10_Issues/               # 作業記録（サイクル別フォルダ）
│   └── cycle-<N>/<ISSUE-ID> <タイトル>.md
├── 20_Knowledge/            # 学びの蓄積（技術ドメイン別）
│   ├── 01_フロントエンド/
│   ├── 02_バックエンド/
│   ├── 03_データベース/
│   ├── 04_インフラ/
│   ├── 05_開発ツール/
│   ├── 06_AI駆動開発/
│   ├── 07_アーキテクチャ/
│   └── 08_CS基礎・Web開発/
└── 90_Templates/
```

新規ノート作成時は適切なカテゴリフォルダに配置し、`category` フロントマターを記録する。
**配置判断は Vault README の「境界判断のルール」と「早見表」を参照する**。

## 入力
- GitHub PR URL（例: `https://github.com/<org>/<repo>/pull/123`）
- イシュートラッカーの URL（紐付くPRを `gh` 経由で自動取得）
- PR番号のみ（リポジトリは cwd から推測）
- 親イシュー URL（複数の子 PR をまとめて咀嚼するパターン）

## 動作

### 1. PR データの取得
```bash
gh pr view <URL or 番号> --json title,body,files,comments,reviews,author,headRefName,baseRefName,mergedAt
gh pr diff <URL or 番号>
```

- リポジトリ判別: URL > cwd の git remote
- イシュー URL の場合: イシュー詳細を取得 → 紐付く PR リンクを抽出
- 親イシューに複数 PR がぶら下がる場合は **全部取得して横断的に咀嚼**
- 取得失敗時はユーザーに確認

### 2. イシュー番号 + サイクルの特定
- ブランチ名 / PRタイトル / PR本文から `[A-Z]+-\d+` を抽出（プロジェクトの ID 形式に応じて調整）
- 作業記録はサイクル別フォルダ `10_Issues/cycle-<N>/` に格納する
- Linear MCP が利用可能なら:
  1. `mcp__linear-server__get_issue` でイシューの `cycleId` を取得
  2. `mcp__linear-server__list_cycles` で該当 `cycleId` の `number` を引く
  3. フォルダ名は `cycle-<number>` (例: `cycle-13`)
- ファイル名規則: `<ISSUE-ID> <タイトル>.md` (例: `PROJ-123 ユーザー登録フォーム実装.md`)
- まず `find 10_Issues -name "<ISSUE-ID>*.md"` で既存ノートを横断検索 (glob で前方一致、タイトル表記揺れに耐える)
- 既存ノートが `source: pr-observation` でなければ「上書きせず別名で作りますか?」と確認
- 新規なら `90_Templates/issue-observation.md` から `10_Issues/cycle-<N>/<ISSUE-ID> <タイトル>.md` を生成
- frontmatter に **必ず `aliases: [<ISSUE-ID>]` を入れる** (これがないと `[[<ISSUE-ID>]]` 形式の wiki link が解決されない)
- サイクル情報が無い・取得できない場合は `10_Issues/` 直下にフォールバック、またはユーザーに確認

### 3. PR 内容の読み込み

PR 本文・diff・ファイル構成・関連リソースを取得する:

- **diff / ファイル構成**: 何が変わったか、どこに新規ファイルが追加されたか、設定ファイル変更の有無
- **PR 本文**: 背景・設計判断・採用構成・スコープ外。ADR / RFC へのリンクがあれば該当ファイルを Read
- **追加修正 PR / hotfix**: マージ後の修正は **本番で初めて気付いた問題** が多く、学びの宝庫
- **レビュー指摘 (あれば)**: AI 前提だと少ないが、あれば優先度高で拾う。一般化した形で、固有名や個人名は記録しない

### 4. ノート構成（重要）

作業記録ノート（`10_Issues/cycle-<N>/<ISSUE-ID> <タイトル>.md`、`source: pr-observation`）は **ジュニアエンジニアの資産化** を目的とする。
細かい実装の詳細は PR を辿れば分かるため、ノート側は **基礎知識と学ぶべきポイント** に絞って書く。

固定セクション（4 つ）:

#### `## 3行サマリ`
箇条書きで 3 項目。流し読みで全体像が掴める粒度に揃える。
- 何のイシュー / どんな対応か
- どんな技術領域に触れたか
- 自分にとって何を学ぶ機会だったか（一言）

#### `## 前提知識`
**そもそも XX とは** — 3年後の自分や別の人が読んでも文脈を再構築できる最低限の説明。
- 専門用語を普通の日本語で言い換える
- 既存の `20_Knowledge/` ノートがあれば `[[Wiki Link]]` を貼って深追いを任せる
- 1〜3 個程度に絞る（PR 本文や ADR の丸写しは NG）

#### `## 実施した内容`
**輪郭だけ**書く（具体ファイル数や行数は基本不要、PR を辿れば分かる）。
- どこに / 何を / なぜ追加・変更したか
- 採用案と却下案がある場合は **一行で対比** する
- ハマりどころは「症状 → 原因 → 解決」を1行で
- チケット文面と実装が乖離した点（コスト・運用負荷・スコープ調整）はここに残す

#### `## 参考・学び`
このイシューで触れた **基礎概念・学ぶべきポイント** だけリスト化する。
- 細かいライブラリの罠は書かない（ググれば出るもの）
- `[[20_Knowledge/...]]` への wiki link + 一言コメントが基本形
- 未作成の概念ノートは「[[XXX]] (要作成)」のように残す
- **「3年後の自分・別プロジェクト・別言語でも使えるか?」** をフィルタにする

#### `## 添付資料`（任意・推奨）
PR の規模が大きい・図解や Before/After 比較が学びを深める場合、**リッチな解説 HTML を添付ファイルとして併置** する。

- 配置: MD と同じ `10_Issues/cycle-<N>/` 配下に `<ISSUE-ID>.html` を置く
- MD からは `[詳細解説](<ISSUE-ID>.html)` でリンク
- HTML 本体の中身は **MD で書くと冗長になる視覚要素** に振る:
  - Hero（タイトル + メタ情報）+ TOC（2 列 grid、ゼロパディング番号）
  - Before/After の 2 カラム比較（`.compare` クラス、pre タグ + bad/good 色分け）
  - ルールごとの `.rule-card`（rule-no chip + Why/How `.meta-row`）
  - `.callout`（`.accent` / `.warn` / `.good`）— Core Idea / Gotcha / Insight
- **デザインは [templates/html-attachment-template.html](../templates/html-attachment-template.html) を必ず使う**:
  - スキル実行時に Read してから書く（記憶や推測で CSS を書かない）
  - CSS 変数・色トークン・フォント（Noto Sans JP）・レイアウト構造は変えない
  - `{{TITLE}}` / `{{KIND}}` / `{{DATE}}` / `{{LEDE}}` を置換、コンテンツセクションを差し込む
  - 重要語は `<span class="u">…</span>` でテラコッタ下線強調（Anthropic.com 風）
- **MD の本文と内容を二重管理しない**: HTML は MD の補完。同じ情報を両方に書かず、MD は要約+リンク、HTML はビジュアル解説に役割分担

> HTML はあくまで「視覚的に理解を助ける添付資料」であり、Source of Truth は MD 側。HTML 単体で完結させる必要はない。

### 5. 知識ノートへの切り出し（最大 5 個）

1つのPRから10個もノートを作らない。**最も学びになる5個**に絞る。

優先順位:
1. ハマりどころ・追加修正 PR から抽出された一般原則
2. PR 本文に明記された設計判断（特にチケットと実装の乖離理由）
3. 非自明な設計パターン
4. レビュー指摘から抽出された原則（あれば）

知識ノートを作る場合:
- 既存ノートがあれば「[[XXX]] に "このプロジェクトでの実例" として追記しますか?」
- 既存に近いものがあれば「[[YYY]] の関連として追加しますか?」

**記録しない**:
- 細かいライブラリの罠・workaround（ググれば出るもの）
- ライブラリ固有の関数仕様

### 6. 未定義リンクの検出
新規ノート作成後、本文中の技術用語を抽出して `20_Knowledge/` を grep し、
ノートが未作成のものを1個ずつ提示して作成を確認する。
No なら飛ばす。

### 7. 出力
最後に作成・更新したファイルのパスを箇条書きで提示。

## ノート構造（`20_Knowledge/`）

**必須**: `## サマリ` (何/なぜ/使い所) / `## 具体的な解説` / `## 担当プロジェクトでの実例` / `## 参考情報`

**オプション**（書くと価値が出る時だけ）:
- `## 似た概念との違い` — 比較対象がある時、比較表で
- `## 使う場面 / 使わない場面` — 過剰適用を防ぐ判断軸

雛形であり、内容に応じて取捨選択する。frontmatter: `mastery: shallow` で開始。
ゴミナレッジを溜めないため、ググれば出る細かい知識は記録しない。

## 記録しないもの
- イシュー / PR本文の丸ごとコピー（URLリンクのみ）
- PR作者以外の個人名（author は frontmatter のみ）
- DB接続情報・トークン・APIキー
- レビュー指摘の感情的・人格的な側面（あれば無視、技術的内容のみ抽出）
- 細かいライブラリ固有の罠（方針外）
