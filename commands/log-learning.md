# log-learning

ジュニアエンジニアとしての成長を支える学習記録スキル。
**主目的は CS基礎・フレームワークの使い方・設計パターンの理解を資産化すること**。
細かいライブラリの罠は記録しない。**3年後の自分にも価値がある知識**だけを残す。

## Vault
`~/obsidian-vault/`（ローカル限定、クラウド同期しない前提）

> 別の場所を使う場合は本ファイルのパスを書き換えるか、`OBSIDIAN_VAULT` 環境変数で参照する運用に変更してください。

```
obsidian-vault/
├── 00_Inbox/                # イシュー紐付けできない記録
├── 10_Issues/               # 作業記録（サイクル別フォルダ）
│   └── cycle-<N>/<ISSUE-ID> <タイトル>.md
├── 20_Knowledge/            # 学びの蓄積（技術ドメイン別）★主役
│   ├── 01_フロントエンド/
│   ├── 02_バックエンド/
│   ├── 03_データベース/
│   ├── 04_インフラ/
│   ├── 05_開発ツール/
│   ├── 06_AI駆動開発/
│   ├── 07_アーキテクチャ/
│   └── 08_CS基礎・Web開発/
└── 90_Templates/            # issue.md, note.md, issue-observation.md
```

## イシュー ID の前提

このスキルは「イシュー管理ツール（Linear / Jira / GitHub Issues 等）の ID を持つ作業」と「持たない作業」の両方に対応する。

- イシュー ID 形式は **プロジェクトごとに異なる**（例: `PROJ-45`, `ABC-123`, `#1234`）
- デフォルトでは `[A-Z]+-\d+` 形式を抽出する
- 別形式を使う場合は本スキル内の正規表現を書き換える

## 何を記録するか（フィルタ）

**記録対象**:
- 新しい設計パターンや概念を理解した
- 既存コードを読んで仕組みを理解した
- フレームワーク/ライブラリの仕様の **なぜ** を理解した
- 設計判断のトレードオフをユーザーと議論した
- CS基礎・Web標準（HTTP, OAuth, DOM 等）の理解が深まった

**記録しない**:
- 細かいライブラリの罠・workaround（ググれば出るもの）
- 1回限りのバグ修正の事実
- タスク・ToDo

→ **「3年後の自分・別プロジェクト・別言語でも使えるか?」** を判断軸にする

## 動作

### 0. 起動時の補強候補チェック（最初に1行で通知）
`/log-learning` 起動直後、`20_Knowledge/` を grep して既存ノートの不備を検出:
- `## 担当プロジェクトでの実例` セクション直下に実コードパス未記録
- `## 参考情報` セクション直下にURLが1つも無い
- `mastery: shallow` のまま放置されているノート

該当があれば1行で「補強候補: ...」と通知。無ければ何も言わない。

### 1. イシュー番号の抽出
- `pwd` のディレクトリ名または git ブランチ名から `[A-Z]+-\d+` を抽出
- 大文字小文字を吸収して `XXX-NNNN` 形式に正規化
- 見つからなければ「どのイシューに紐付けますか? (なしなら Inbox に保存)」と確認

### 2. サイクル（スプリント）の特定
- 作業記録はサイクル別フォルダ `10_Issues/cycle-<N>/` に格納する
- Linear MCP が利用可能なら:
  1. `mcp__linear-server__get_issue` でイシューの `cycleId` を取得
  2. `mcp__linear-server__list_cycles` で該当 `cycleId` の `number` を引く
  3. フォルダ名は `cycle-<number>` (例: `cycle-13`)
- 既存の `cycle-<N>/<ISSUE-ID> <タイトル>.md` を `find /path/to/10_Issues -name "<ISSUE-ID>.md"` で先に探す（既存があればそちらを使う）
- サイクル情報が無い・取得できない場合は `10_Issues/` 直下にフォールバック、またはユーザーに確認

### 3. 作業記録ノートの準備
- ファイル名規則: `<ISSUE-ID> <タイトル>.md` (例: `PROJ-123 ユーザー登録フォーム実装.md`)
- まず `find 10_Issues -name "<ISSUE-ID>*.md"` で既存ノートを探す (タイトル部分の表記揺れに耐えるため glob で前方一致)
- イシュー番号 + サイクル番号があれば `10_Issues/cycle-<N>/<ISSUE-ID> <タイトル>.md` を対象に
  - 無ければ `90_Templates/issue.md` をコピーして frontmatter を埋める
  - frontmatter に **必ず `aliases: [<ISSUE-ID>]` を入れる** (これがないと `[[<ISSUE-ID>]]` 形式の wiki link が解決されない)
  - あれば必ず Read してから Edit（セクション構造を把握）
- イシュー番号が無ければ `00_Inbox/YYYY-MM-DD-スラッグ.md` に新規作成

### 4. 作業記録ノートの構成

作業記録ノートは **ジュニアエンジニアの資産化** を目的とする。
細かい実装の詳細はコード / git 履歴を辿れば分かるため、ノート側は **基礎知識と学ぶべきポイント** に絞って書く。

固定セクション（4 つ）:

#### `## 3行サマリ`
箇条書きで 3 項目。流し読みで全体像が掴める粒度に揃える。
- 何のイシュー / どんな対応か
- どんな技術領域に触れたか
- 自分にとって何を学ぶ機会だったか（一言）

#### `## 前提知識`
**そもそも XX とは** — 3年後の自分や別の人が読んでも文脈を再構築できる最低限の説明。
- 専門用語を普通の日本語で言い換える
- 既存の `20_Knowledge/` ノートがあれば `[[Wiki Link]]` を貼って深追いを任せる
- 1〜3 個程度に絞る（公式ドキュメントや ADR の丸写しは NG）

#### `## 実施した内容`
**輪郭だけ**書く（具体ファイル数や行数は基本不要）。
- どこに / 何を / なぜ追加・変更したか
- 採用案と却下案がある場合は **一行で対比** する
- ハマりどころは「症状 → 原因 → 解決」を1行で

#### `## 参考・学び`
このイシューで触れた **基礎概念・学ぶべきポイント** だけリスト化する。
- 細かいライブラリの罠は書かない（ググれば出るもの）
- `[[20_Knowledge/...]]` への wiki link + 一言コメントが基本形
- 未作成の概念ノートは「[[XXX]] (要作成)」のように残す
- **「3年後の自分・別プロジェクト・別言語でも使えるか?」** をフィルタにする
- 各項目の先頭に `(YYYY-MM-DD)` を付けると追記履歴が残る

#### `## 添付資料`（任意）
図解や Before/After 比較が学びを深める場合、**リッチな解説 HTML を添付ファイル** として併置する。
- 配置: 同じフォルダに `<ISSUE-ID>.html`、MD から `[詳細解説](<ISSUE-ID>.html)` でリンク
- 役割分担: MD は要約 + wiki link、HTML は視覚解説（hero / TOC / Before-After 2 カラム / `.rule-card` Why-How / `.callout`）
- **デザインは [templates/html-attachment-template.html](../templates/html-attachment-template.html) を必ず使う**: スキル実行時に Read してから書く。CSS 変数・色トークン・フォント（Noto Sans JP）・レイアウト構造は変えない。重要語は `<span class="u">…</span>` でテラコッタ下線強調
- Source of Truth は MD 側、HTML は補完（同じ情報を両方に書かない）

```markdown
## 参考・学び
- (2026-05-18) CSS の優先順位の仕組みを理解 → [[CSS Specificity と Cascade]]
```

### 5. 知識ノートの作成判定

ユーザーから抽出した学びについて、**「3年後の自分・別プロジェクト・別言語でも使えるか?」** を問う:
- Yes → `20_Knowledge/` への切り出しを提案
- No → 作業記録ノートに留める（記録対象外なら記録しない選択肢も）

知識ノート作成時:
- 既存の `20_Knowledge/` を grep して類似ノートが無いか確認
- 既存なら追記提案、無ければ `90_Templates/note.md` から新規作成
- **配置先カテゴリは Vault の README の「境界判断のルール」と「早見表」に従う**
- frontmatter に `category: XX_カテゴリ名` を記録

### 6. カテゴリ判断のルール

```
Q1. ライブラリ・ツールが変わっても通用する原則か?
    Yes → 07_アーキテクチャ または 08_CS基礎・Web開発
    No  → Q2

Q2. 主にどのレイヤーで使うか?
    ブラウザ実行 → 01_フロントエンド
    サーバー実行 → 02_バックエンド
    データ永続化 → 03_データベース
    クラウド基盤 → 04_インフラ
    開発・運用   → 05_開発ツール
    AI ツール    → 06_AI駆動開発

Q3. 迷う? → 07_アーキテクチャ
```

迷う場合は **Vault README の早見表** を参照し、似たパターンに揃える。

### 7. ノートの構造（雛形、内容に応じて最適化）

`20_Knowledge/` のノートは **必要な情報だけ書く**（ゴミナレッジを溜めない）。
ググれば出る細かい知識ではなく、ソフトウェアエンジニアとして一般的に知っておくべきことだけ記録する。

**必須セクション**:
- `## サマリ` — **何 / なぜ / 使い所** の3点を1行ずつ
- `## 具体的な解説` — 仕組み・規則。**抽象的になる時は必ずサンプルコード・図・具体例を添える**
- `## 担当プロジェクトでの実例` — 関連イシュー + 実コードへのパス
- `## 参考情報` — 公式ドキュメント / MDN / 書籍 URL

**`## 具体的な解説` のコツ**:
- 記号的な記法（`(1,0,0,0)` 等）だけで終わらせない
- 実際のコード例 → 結果、の順で示す
- 表が有効な場合は表を使う
- 「これとこれが衝突したらどうなるか」を具体例で見せる

**オプションセクション**（書くと価値が出る時だけ書く）:
- `## 似た概念との違い` — 比較対象がある時。比較表で書く
- `## 使う場面 / 使わない場面` — 過剰適用を防ぐ判断軸が必要な時

**全部書く義務感を持たない**。内容に応じて取捨選択する。

**`## 担当プロジェクトでの実例` の埋め方**:
1. プロジェクト内のソースツリーから、その概念が使われている実装を grep / glob で探す
2. 候補を1〜3個提示し、ユーザーに「この実装で合ってますか?」と確認
3. 確認できたら `[ファイル名](絶対パス:行番号)` の形式で記録

### 8. 理解度の自己チェック（書き終わりに1問だけ）

ノート作成完了時に1問だけ問う:

> 「この概念、`## サマリ` を見ずに自分の言葉で説明できますか?」

回答に応じてフロントマターを更新:
- 説明できる → `mastery: learned`
- 説明できない → `mastery: shallow`（初期値のまま）

### 9. 未定義リンクの検出（学びの連鎖を生む）

新規ノート作成後、本文中の技術用語を抽出して `20_Knowledge/` を grep し、
ノートが未作成のものを検出して1個ずつ提示。

```
このノートに以下の用語が出てきますが、ノートが未作成です:
- Reconciliation
- Hydration

これらを 20_Knowledge/01_フロントエンド/ に新規作成しますか?（1個ずつ確認）
```

ユーザーが Yes と言ったものだけ作成。No なら飛ばす。

### 10. タグ候補の提案（軽量）
`tags: [concept, react, frontend]` のように分野軸で。
厳密化は不要。

### 11. _index.md の運用（任意）
`20_Knowledge/XX_カテゴリ/_index.md` がある場合、新規ノート作成時に追記提案。
無くてもOK。`_index.md` はノートが5個超えたら作る候補。

### 12. 出力
最後に作成/更新したファイルのパスを箇条書きで提示。

## 編集ルール
- 既存ファイルは**必ず Read してから Edit**
- 作業記録ノートのセクション構造は固定（## 3行サマリ / ## 前提知識 / ## 実施した内容 / ## 参考・学び / ## 関連）
- 参考・学び項目の先頭に `(YYYY-MM-DD)` を付けると追記履歴が残る
- `updated` フロントマターを当日に更新

## トリガー（自発的に提案する場面）

**強いシグナル**（積極的に提案）:
- 新しい設計パターンや概念を初めて知った
- 既存コードを読んで「こういう設計か」と理解した
- フレームワーク/ライブラリの仕様の **なぜ** を理解した
- 設計判断のトレードオフを議論した
- ユーザーが「なるほど」「面白い」「いいやり方」等のシグナルを発した

**弱いシグナル**（記録対象外、提案しない）:
- 単純なライブラリ固有のバグ・workaround
- ググれば解決する初歩的なエラー

## 禁止
- ユーザー確認なしの新規ノート作成
- DB接続情報・JWT・APIキー・.env の中身の記録
- イシュー本文のコピー（URLリンクのみ）
- セッション内で既に「No」と言われた事項の再提案
- 細かいライブラリ固有の罠を記録すること（方針外）
