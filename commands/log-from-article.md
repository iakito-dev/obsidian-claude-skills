# log-from-article

Zenn / Qiita / Medium / 個人ブログ / Speaker Deck / SlideShare / GitHub README など、
**外部の文章コンテンツ** を読んで Obsidian に学びを残すスキル。

**Zettelkasten の「Literature Note」に相当**: 外部材料を自分の言葉で咀嚼し、
そこから抽出できる概念は `20_Knowledge/` の概念ノートに昇華する。

## Vault
`~/obsidian-vault/40_Articles/`（外部記事ノートの格納先）

## 入力
- 記事 URL（Zenn / Qiita / Medium / Speaker Deck / GitHub README / ブログ等）
- PDF / スライドの URL
- 「これ読んで」と渡される本文テキスト

## 動作

### 1. コンテンツ取得
- URL なら WebFetch で取得（認証必要なら諦めて手動入力を依頼）
- 長文なら要点抽出を Claude が行う

### 2. ノート作成準備
- ファイル名: `YYYY-MM-DD-スラッグ.md` 形式
- スラッグは記事タイトルから生成（日本語OK、半角スペースは `-` に）
- 例: `2026-05-18-obsidian-pkm-best-practices.md`

### 3. ノート内容の生成（5セクション）

#### a. 3行サマリ
何の記事か、何が書かれているかを **3 行程度の自然な日本語**で書く。
- 各行は 50〜70 字程度。長文化させない
- 構造化（「何 / 主張 / 得たもの」のようなラベル付け）はしない
- ただの文章として読める形にする

#### b. 内容の解説
記事の主張・論点を**自分の言葉で咀嚼**して再構成する。
- **コピペ禁止**。記事の専門用語（「危険度マップ」「思考ジャーナル」のような独自の呼称）はそのまま使わず、**普通の日本語で言い換える**
- 例: 「文脈を選択的に読み込む」→「必要な情報だけを渡している」
- 例: 「思考ジャーナルで連続性を保証」→「前日に考えたことを引き継ぐので、毎晩バラバラに改善するのではない」
- 重要な引用は 1〜2 個まで `>` ブロックで残してよい
- **記事には書かれていない前提・暗黙のメッセージ**を読み取って明示する。
  例: 「成立する前提: E2E テスト / Design Doc / 規約整備が整っていること。これがないと運用できないのに記事は強調していない」

#### c. 参考になる点
- 自分の業務・学習に**取り入れられそうな観点**
- 既存の `20_Knowledge/` に追記すべき内容があれば提案
- 新規概念ノート化の候補があれば提示（ユーザー確認後に作成）

#### d. 疑問・反論
記事の主張に対して:
- 納得しなかった部分
- 不明な前提・データ（コストや工数が示されていない等）
- 別の意見・反例・トレードオフ
- 触れられていない懸念（教育機会の損失、組織前提の依存など）

→ **鵜呑みを防ぐ批判的読解**の記録。

#### e. 出典
URL / 著者を frontmatter と本文の両方に残す。

### 4. セルフレビュー → 提示

ユーザーに見せる前に、自分で以下をチェック:
- [ ] 記事の独自用語をそのまま使っていないか?（使っていれば普通の日本語に言い換える）
- [ ] 3行サマリが本当に 3 行か?（実質 3 段落になっていないか）
- [ ] 内容の解説が記事のコピペになっていないか?
- [ ] 「記事に書かれていない前提」を 1 つ以上指摘できているか?
- [ ] 疑問・反論が記事の主張に対する**鋭い**ものか?（トートロジーになっていないか）
- [ ] **生身の人間が読んで疲れない**文章になっているか?

不合格項目があれば直してから提示する。

### 5. ユーザーへの提示・確認の方針

デフォルトは **確認なしで書き込む**（ユーザーが信頼している前提）。
セルフレビューチェックリストをパスしていれば、そのままノートを作成する。

例外的に確認するケース:
- 概念ノート（`20_Knowledge/`）を 3 つ以上同時に作る場合 → 数を減らすか確認
- カテゴリ判定が難しいケース → どこに置くか確認
- ユーザーが「先に見せて」と明示した場合

→ 通常は外部記事ノート 1 個 + 概念ノート 0〜3 個を**一気に書き込む**。
完成後にファイルパスを箇条書きで提示する。

### 6. 概念ノートへの昇華

「参考になる点」セクションで概念ノート化を提案する場合:
- 既存の `20_Knowledge/XX_カテゴリ/` を grep して類似ノートを探す
- 既存があれば「[[既存ノート]] に追記しますか?」
- 無ければ「新規 `20_Knowledge/XX_カテゴリ/タイトル.md` を作りますか?」
- Vault README の「境界判断のルール」と「早見表」に従ってカテゴリ判定

概念ノートを作る場合は **必ず以下を含める**:
- 標準テンプレ（サマリ / 具体的な解説 / 担当プロジェクトでの実例 / 参考情報）を使う
- 「具体的な解説」には**コード例 / yaml / 表など**を最低 1 つ入れる（抽象論で終わらせない）
- 「似た概念との違い」を比較表で書く（区別すべき近接概念がある時）
- **双方向リンク**:
  - 概念ノート側の「担当プロジェクトでの実例」または「参考情報」に `[[40_Articles/記事ノート名]]` を貼る
  - 外部記事ノート側の「参考になる点」に `[[概念ノート名]]` を貼る

### 7. 出力
最後に作成・更新したファイルのパスを箇条書きで提示。

## 文体ガイド（生身の人間が読みやすく）

**OK な書き方**:
- 「AI を 3 つに分けて専門家にする」
- 「AI が毎晩自分で賢くなる仕組み」
- 「必要な情報だけを渡している」
- 「PR レビューは新人がシニアの考え方を学ぶ場でもある」

**NG な書き方**（硬すぎる・記事のコピペ）:
- 「文脈を選択的に読み込む構造」
- 「危険度マップを自動更新するメカニズム」
- 「思考ジャーナルで連続性を保証」
- 「役割の境界を明確に画定することで両者の能力を最大化」

**専門用語は使ってよい**: AI レビュー、CLAUDE.md、skill、E2E テスト、Design Doc、PR、リファクタリング、アーキテクチャ などはそのまま使う。問題は**専門用語ではなく硬い言い回し**。

## ノート構造（`40_Articles/`）

frontmatter:
```yaml
tags: [external]
type: article | slide | book | video | talk
source: URL
author: 著者名
date_read: 読んだ日付
created: 作成日
```

セクション:
- `## 3行サマリ` — 何の記事 / 主張 / 得たもの
- `## 内容の解説` — 自分の言葉で咀嚼した記事の論点
- `## 参考になる点` — 取り入れられる観点 + 20_Knowledge への接続
- `## 疑問・反論` — 批判的読解の記録
- `## 出典` — URL / 著者

## 記録しないもの
- 記事本文の丸ごとコピー（要点を咀嚼したものだけ）
- 機密情報を含む内部記事の本文
- ググれば出る範囲の細かい How-to

## トリガー
ユーザーが明示的に呼ぶ前提。
URL を渡された時のみ動作。Claude が自発的に提案するシーンは無い。
